- **The core fix:** Phase 14 still computed `net_pay` as `base_salary тИТ deductions`, so 0 present days still netted most of a month's pay (absence was "one more deduction against an assumed full salary," not "you weren't paid"). Net pay is now **earnings-based**: `earned_base_pay = (present_days + half_days├Ч0.5 + paid_leave_days + holiday_days + compensation_days) ├Ч per_day_rate` (`per_day_rate = base_salary ├╖ 30`). **Absent and unpaid-leave days are simply excluded from earned_base_pay тАФ identical treatment, no separate punitive deduction; not being paid for the day is the only consequence.** Late-coming remains a separate penalty *on top of* days actually worked: `late_count ├Ч late_deduction_percent% ├Ч per_day_rate`. Compensation days (Sunday clock-ins) are paid at the normal per-day rate тАФ same weight as present, no overtime premium (confirmed with Umang). `net_pay = earned_base_pay + incentive + bonus + other_addition тИТ late_deduction_total тИТ other_deduction + reimbursement_total`.

### Phase 14 тАФ Salary-proportional deductions (2026-07-17, on `main`)
`bun run typecheck` / `lint` / `build` / `bun test` (45/45) all clean. Live-verified against the dev server + seeded DB with a hand-computed worked example (base salary 50,000, late% 20, 2 late + 1 half-day + 1 unpaid-leave + 12 absent days тЖТ `standard_deduction_total` 23,166.67, matching `(2├Ч0.20 + 0.5├Ч1 + 1├Ч1 + 1├Ч12) ├Ч (50000/30)` exactly); test data (attendance records, leave request, extra payroll-config row, team's `expectedStartTime`) cleaned up afterward.
- **New formula (тЪая╕П real money-math change, confirmed with Umang):** deductions are now proportional to each employee's own `base_salary`, not flat company-wide rupee amounts. `per_day_rate = base_salary ├╖ 30` (fixed 30-day-month convention). Late = a **configurable percentage** of one day's pay per occurrence (`PayrollConfig.lateDeductionPercent`); half-day = a **fixed** 50% of a day's pay; unpaid leave = a **fixed** full day's pay; **absent days now deduct a full day's pay too тАФ previously absent was the one attendance state that deducted nothing at all.** Half-day/unpaid-leave/absent are fixed fractions (mathematical facts, not policy), so they're no longer separately configurable тАФ `PayrollConfig` shrank to the single `late_deduction_percent` knob. All in `lib/payroll/calc.ts` (`computePerDayRate`, rewritten `computeStandardDeductionTotal`).
- **Data-source consolidation:** payroll generation now sources half-day/unpaid-leave/absent counts from the holiday- and Sunday-compensation-aware `lib/attendance/monthly-breakdown.ts` (built Phase 13) instead of the simpler flag-counting in `lib/attendance/summary.ts`; only `lateCount` (+ its `lateTrackingUnavailable` note) still comes from the latter, since nothing else computes late-arrival.
- **Schema (тЪая╕П shared, migration `20260717060000_late_deduction_percent_and_absent_count`):** `PayrollConfig` тАФ dropped `lateDeductionFlat`/`halfDayDeductionFlat`/`unpaidLeaveDeductionFlat`, added `lateDeductionPercent` (existing row backfilled to `0%` тАФ a safe default, no surprise deduction until Admin explicitly sets a rate). `Payslip` тАФ added `absentCount` (persisted alongside the existing counts for auditability/reproducibility).

### Phase 12 тАФ Forgot/reset password (2026-07-16, on `main`)
Build + lint + typecheck clean. Live-verified against the dev server + seeded DB with a throwaway user pointed at a real inbox (a real `POST` to Brevo's API succeeded, not mocked); test data cleaned up afterward.
- **Flow:** `POST /api/v1/auth/forgot-password` (`{ email }`, public) always returns the same generic response regardless of whether the account exists тАФ no enumeration signal. If it does exist: any previous unused token for that user is invalidated, a random 32-byte token is generated, its **sha256 hash** (never the raw token) is stored in a new `PasswordResetToken` row with a **15-minute expiry**, and an email is sent via Brevo with a link to `/reset-password?token=...`. `POST /api/v1/auth/reset-password` (`{ token, new_password }`, public) validates the token (unused + unexpired), enforces the existing `checkPasswordStrength` policy, and тАФ matching change-password's precedent тАФ **bumps `tokenVersion`** (revokes every outstanding session) and marks the token used; it does **not** auto-login, the user returns to `/login`. Both routes rate-limited (`checkRateLimit`, 5/15min and 10/15min respectively) and audited (`auth.forgot_password_requested`, `auth.password_reset`).
- **Schema (тЪая╕П shared file, migration `20260716130208_add_password_reset_tokens`):** new `PasswordResetToken` table (`userId`, unique `tokenHash`, `expiresAt`, `usedAt`) + `User.resetTokens` back-relation. No changes to existing `User` columns.
