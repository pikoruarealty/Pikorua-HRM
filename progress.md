### AI task generation (2026-07-14, post-integration — built directly on `main`)
New feature (user request): an LLM breaks a project brief into its task hierarchy. `POST /api/v1/work-units/:id/generate-tasks` reads the WorkUnit's brief (`body.description` or the new stored `WorkUnit.description`) and asks Groq to propose SubUnits + WorkItems. **Default = draft** (returns the proposal, no DB writes — the Lead reviews); **`persist:true` + `defaultAssigneeId`** creates the whole tree in one transaction, assigned to that employee (same team-membership rule as the manual work-item route). Mode auto-derives from the department via `DepartmentLabel`: **atomic** (`taskPoints`) for tech, **metric** (`targetValue`, current period) for sales/BD. RBAC mirrors the manual sub-unit/work-item routes (Admin/HR or owning Lead). Infra: `lib/ai/groq.ts` (fetch-based Groq client, OpenAI-compatible chat, JSON mode, 30s timeout — **no new dependency**, so `package.json` untouched) + `lib/ai/task-generation.ts` (prompt + zod-validation + clamping, `MAX_SUB_UNITS`/`MAX_ITEMS_PER_SUB_UNIT`). Model default `llama-3.3-70b-versatile`, override via `GROQ_MODEL`. **Schema (shared, flagged):** added `WorkUnit.description` (nullable) + migration `20260714171037_add_workunit_description`, cascaded into work-units create/patch. `.env.example` documents `GROQ_API_KEY`/`GROQ_MODEL`. **Live-verified** against real Groq + seeded DB: atomic draft + persist (rows confirmed via nested GET), metric draft (Campaign/Target Segment/Call + `targetValue`), and edges (persist-without-assignee 422, non-lead 403, missing-description 422). `bun run build` clean; test data cleaned up; dev server stopped.
