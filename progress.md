### Phase 9 — FCM push notifications (2026-07-15, on `main`)
Build + lint + typecheck + 43 unit tests all clean. - **Single chokepoint:** `pushNotification()` in `lib/notifications/push.ts` (flagged, not on the shared list but the established reusable-service pattern) now does the in-app `notifications` insert **and** fans out to FCM — every existing and future call site (leave approve/reject/override, meeting reminders, birthday/anniversary, EOD summary, recognition) gets push for free, no per-call-site changes needed. Push send is fire-and-safe (never blocks or fails the underlying mutation), matching the `audit()`/`createLogger()` contract established in Phase 8.
- **Schema:** new `push_tokens` table (`userId`, unique `token`) + `User.pushTokens` back-relation (migration `20260715122621_add_push_tokens`).
- **Server (`lib/notifications/fcm.ts`):** lazy `firebase-admin` app init from `FIREBASE_ADMIN_*` env vars; missing/placeholder config degrades to a no-op (logged at debug) rather than throwing — same posture as the optional GROQ/S3 config. `sendPushToUser()` sends to every token for a user via `Promise.allSettled`, and **auto-prunes tokens FCM reports as unregistered/invalid** (reads the error's `.code` directly — note: firebase-admin's `FirebaseError` puts `code` at the top level, not nested under `.errorInfo` — this was caught and fixed during live verification, not assumed).
- **API (self-only):** `POST`/`DELETE /api/v1/notifications/push-token` — a browser can only register/unregister its own token against the logged-in user; upserts on the unique token (handles a shared/kiosk browser re-registering under a different account) and cross-user delete attempts are a silent no-op (deleteMany with a non-matching where).
- **Client:** `lib/firebase/client.ts` (lazy Firebase app/Messaging init, guarded for SSR), `lib/firebase/messaging-client.ts` (permission request → SW registration → `getToken` → register with the server; foreground messages shown via a manual `Notification` since FCM doesn't auto-show those), `public/firebase-messaging-sw.js` (background push handler — reads its Firebase config from the SW registration URL's query string, since static files can't read `NEXT_PUBLIC_*` build-time env vars).
- **UI — opt-in only, never auto-prompted:** new "Push notifications" card on `/settings` (`components/settings/push-notifications-toggle.tsx`) below Account Security, for every role. Shows unsupported/unconfigured/blocked/on/off states; Enable/Disable is a deliberate click, matching the browser-permission-should-be-a-conscious-grant principle from the original plan.
- **Docs:** `.env.example` (9 new Firebase vars, all documented as optional/degrade-gracefully), README "Push notifications" section, `docs/` left alone (this is infra, not a new API_SPEC endpoint category beyond the 2 push-token routes — noted inline where relevant).
