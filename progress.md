### Phase 15 — Earned-pay formula + live preview (2026-07-17, on `main`, same-day follow-up to Phase 14)
`bun run typecheck` / `lint` / `build` / `bun test` (47/47) all clean. Live-verified against the dev server + seeded DB: reproduced the exact bug reported (Tech Employee, July 2026, 0 present / 15 absent days out of 15 elapsed) — `POST /payslips/preview` now returns `earned_base_pay: 0, net_pay: 0` (was ₹25,000 under Phase 14's formula); a mixed scenario (1 present + 2 half-day + 1 late + 12 absent, 25% late rate) hand-verified to `earned_base_pay 3333.33, late_deduction_total 416.67, net_pay 3916.67`, and confirmed the actual `POST /payslips/generate` persists identical numbers to what the preview showed. Test data (attendance records, team `expectedStartTime`, extra payroll-config rows, a generated payslip) cleaned up afterward; also cleaned up two stray `payroll_config` rows left over from Phase 14's own verification pass.
- **The core fix:** Phase 14 still computed `net_pay` as `base_salary − deductions`, so 0 present days still netted most of a month's pay (absence was "one more deduction against an assumed full salary," not "you weren't paid"). Net pay is now **earnings-based**: `earned_base_pay = (present_days + half_days×0.5 + paid_leave_days + holiday_days + compensation_days) × per_day_rate` (`per_day_rate = base_salary ÷ 30`). **Absent and unpaid-leave days are simply excluded from earned_base_pay — identical treatment, no separate punitive deduction; not being paid for the day is the only consequence.** Late-coming remains a separate penalty *on top of* days actually worked: `late_count × late_deduction_percent% × per_day_rate`. Compensation days (Sunday clock-ins) are paid at the normal per-day rate — same weight as present, no overtime premium (confirmed with Umang). `net_pay = earned_base_pay + incentive + bonus + other_addition − late_deduction_total − other_deduction + reimbursement_total`.
- **Schema (⚠️ shared, migration `20260717080000_earned_base_pay_and_full_day_counts`, no data loss — 0 payslip rows existed):** `Payslip` gained `presentCount`, `paidLeaveCount`, `holidayCount`, `compensationCount`, `earnedBasePay` (every count that now feeds the formula, stored for audit reproducibility like the existing counts); **renamed** `standardDeductionTotal` → `lateDeductionTotal` since it now only ever represents the late penalty (keeping the old name would misleadingly imply half-day/unpaid/absent still subtract from it).
- **New shared module `lib/payroll/payslip-preview.ts`** (`computePayslipPreview`): the employee lookup, `getEffectivePayrollConfig`, attendance summary/breakdown calls, and the cross-track `getApprovedReimbursementTotal`/`getEmployeeOfMonthStatus` calls (same `NotImplementedError` handling as before) — used by **both** `POST /payslips/generate` (persists) and the **new `POST /payslips/preview`** (Admin/HR, read-only, no persistence, no duplicate-payslip check — safe to call repeatedly), so the two can never drift apart.
- **UI — the actual ask:** the Generate Payslip screen (`payslips-screen.tsx`) now auto-computes and shows the **full salary breakdown and the resulting net payable amount** as soon as an employee (+ month/year) is picked — before Generate is ever clicked — via one debounced (~400ms) call to the new preview endpoint that also re-fires on every manual-field edit (incentive/bonus/other addition/deduction), replacing the old two-fetch effect (a separate attendance-summary call + a separate employee-of-month-status call). `payslip-detail.tsx` updated to the renamed/new fields, with the day-count breakdown now read directly off the stored payslip (no longer a second re-derived fetch, since the counts are genuinely persisted now).
- **Docs updated:** PRD.md §5.2 + glossary, SCHEMA.md §4 (`payslips` table), API_SPEC.md §6 (new `POST /payslips/preview` row) — reworded from "base minus deductions" to earnings-based proration.

### Phase 14 — Salary-proportional deductions (2026-07-17, on `main`)
`bun run typecheck` / `lint` / `build` / `bun test` (45/45) all clean. Live-verified against the dev server + seeded DB with a hand-computed worked example (base salary 50,000, late% 20, 2 late + 1 half-day + 1 unpaid-leave + 12 absent days → `standard_deduction_total` 23,166.67, matching `(2×0.20 + 0.5×1 + 1×1 + 1×12) × (50000/30)` exactly); test data (attendance records, leave request, extra payroll-config row, team's `expectedStartTime`) cleaned up afterward.
- **New formula (⚠️ real money-math change, confirmed with Umang):** deductions are now proportional to each employee's own `base_salary`, not flat company-wide rupee amounts. `per_day_rate = base_salary ÷ 30` (fixed 30-day-month convention). Late = a **configurable percentage** of one day's pay per occurrence (`PayrollConfig.lateDeductionPercent`); half-day = a **fixed** 50% of a day's pay; unpaid leave = a **fixed** full day's pay; **absent days now deduct a full day's pay too — previously absent was the one attendance state that deducted nothing at all.** Half-day/unpaid-leave/absent are fixed fractions (mathematical facts, not policy), so they're no longer separately configurable — `PayrollConfig` shrank to the single `late_deduction_percent` knob. All in `lib/payroll/calc.ts` (`computePerDayRate`, rewritten `computeStandardDeductionTotal`).
- **Data-source consolidation:** payroll generation now sources half-day/unpaid-leave/absent counts from the holiday- and Sunday-compensation-aware `lib/attendance/monthly-breakdown.ts` (built Phase 13) instead of the simpler flag-counting in `lib/attendance/summary.ts`; only `lateCount` (+ its `lateTrackingUnavailable` note) still comes from the latter, since nothing else computes late-arrival.
- **Schema (⚠️ shared, migration `20260717060000_late_deduction_percent_and_absent_count`):** `PayrollConfig` — dropped `lateDeductionFlat`/`halfDayDeductionFlat`/`unpaidLeaveDeductionFlat`, added `lateDeductionPercent` (existing row backfilled to `0%` — a safe default, no surprise deduction until Admin explicitly sets a rate). `Payslip` — added `absentCount` (persisted alongside the existing counts for auditability/reproducibility).
- **API:** `PUT /payroll/config` body is now `{ late_deduction_percent, effective_from }`. `POST /payslips/generate` response gained `absentCount`.
- **UI:** `/payroll/config` now shows/edits a single "late deduction (%)" rate plus explanatory copy on the fixed half-day/unpaid-leave/absent fractions. The payslip generate-form preview and payslip detail page's deduction breakdown both gained an "absent" figure alongside late/half-day/unpaid-leave.

### Phase 12 — Forgot/reset password (2026-07-16, on `main`)
Build + lint + typecheck clean. Live-verified against the dev server + seeded DB with a throwaway user pointed at a real inbox (a real `POST` to Brevo's API succeeded, not mocked); test data cleaned up afterward.
- **Flow:** `POST /api/v1/auth/forgot-password` (`{ email }`, public) always returns the same generic response regardless of whether the account exists — no enumeration signal. If it does exist: any previous unused token for that user is invalidated, a random 32-byte token is generated, its **sha256 hash** (never the raw token) is stored in a new `PasswordResetToken` row with a **15-minute expiry**, and an email is sent via Brevo with a link to `/reset-password?token=...`. `POST /api/v1/auth/reset-password` (`{ token, new_password }`, public) validates the token (unused + unexpired), enforces the existing `checkPasswordStrength` policy, and — matching change-password's precedent — **bumps `tokenVersion`** (revokes every outstanding session) and marks the token used; it does **not** auto-login, the user returns to `/login`. Both routes rate-limited (`checkRateLimit`, 5/15min and 10/15min respectively) and audited (`auth.forgot_password_requested`, `auth.password_reset`).
- **Schema (⚠️ shared file, migration `20260716130208_add_password_reset_tokens`):** new `PasswordResetToken` table (`userId`, unique `tokenHash`, `expiresAt`, `usedAt`) + `User.resetTokens` back-relation. No changes to existing `User` columns.
- **Email infra (new, `lib/email/`):** `brevo.ts` — fetch-based client (no SDK, matches `lib/ai/groq.ts`'s style, `apps/web/package.json` untouched) calling Brevo's `v3/smtp/email`. **Degrades gracefully**: missing/placeholder `BREVO_API_KEY` logs at debug and no-ops rather than throwing (verified live: forgot-password still returns its normal 200 with the key unset). `templates/password-reset.ts` — a plain inline-styled HTML email, no template-engine dependency.
- **UI:** `app/(auth)/forgot-password/page.tsx` (email form, shows the same generic message regardless of API response, matching the no-enumeration guarantee client-side too) and `app/(auth)/reset-password/page.tsx` (reads `?token=`, reuses the existing `PasswordField` show/hide component from Phase 11, confirm-password match check). "Forgot password?" link added to `/login`.
- **Env:** `APP_BASE_URL` (builds the email's reset link; `https://` is assumed if no scheme is given — normalized in code) and `BREVO_API_KEY`/`BREVO_SENDER_EMAIL`/`BREVO_SENDER_NAME` (sender verified in Brevo: `pikoruaweb@gmail.com` / "PIKORUA"). Documented in `.env.example`.
- **Live-verified:** real Brevo send succeeded (200 from their API); weak-password 422; valid reset 200 → old password rejected, new password logs in; token reuse 422 (invalid/expired); expired token 422 (backdated `expiresAt` in a throwaway script); rate limit 429 on the 6th forgot-password call from the same IP within the window; Brevo-unconfigured case still returns the normal 200 without throwing.

The first attempt at a fix went **data-only** so the SDK stayed quiet and our SW handler was the sole renderer — that then showed a bare `"Pikorua HRM"` with no body, because it moved rendering into the one file a browser may serve from a stale cache, and an old SW reads `payload.notification`, which data-only messages don't have. **Current design (reversed, 2026-07-16):** send `notification:` (+ `webpush.notification.icon` / `webpush.fcmOptions.link`) and register **no** `onBackgroundMessage` and no custom `notificationclick` — the SDK renders and handles clicks, exactly once, and stale-worker-proof since nothing depends on SW code being current. `firebase-messaging-sw.js` is now init-only; **don't re-add a background handler** or duplicates return.
